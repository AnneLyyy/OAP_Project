Лабораторна робота №5. Уразливості і захист

Проєкт Project_OAP

Проєкт Project_OAP — це вебзастосунок для роботи з подіями. Він дозволяє створювати, переглядати, редагувати та видаляти події, а також працювати з додатковими функціями для аналізу даних.

Основний функціонал проєкту включає CRUD для подій, пошук подій за назвою, сортування, пагінацію, фільтр за датою, перегляд топ-3 найбільших подій, підрахунок кількості подій, перегляд подій разом з користувачами, а також формування та відображення статистики у таблиці TaskStats.

У лабораторній роботі №5 функціонал проєкту використано для демонстрації та виправлення типових уразливостей вебзастосунків: SQL Injection, XSS, Broken Access Control / IDOR та Security Misconfiguration.

Поточний користувач

Для навчального сценарію використовується демо-ідентифікація через HTTP-заголовок X-Demo-UserId.

Приклад заголовка:

X-Demo-UserId: user-1

У проєкті передбачено тестових користувачів:

user-1 — Анна
user-2 — Богдан
user-3 — Марія

Якщо заголовок відсутній або користувача не існує, backend повертає відповідь 401 Unauthorized у єдиному форматі помилки.

Короткий опис ризиків і виправлень

Для сценарію SQL Injection основний ризик полягає в тому, що користувацький ввід може впливати на SQL-запит. Наслідком може бути отримання зайвих даних або помилка бази даних. Для виправлення використано параметризовані запити та allowlist для сортування.

Для сценарію XSS ризик полягає в тому, що дані користувача можуть інтерпретуватися браузером як HTML або JavaScript. Це може призвести до зміни DOM або виконання небажаного коду. Для виправлення використано безпечне відображення через textContent.

Для сценарію IDOR ризик полягає в тому, що користувач може підставити чужий id у запит. Наслідком може бути читання, зміна або видалення чужої події. Для виправлення додано перевірку власника ресурсу на backend.

Для сценарію Security Misconfiguration ризик полягає в небезпечних або неповних налаштуваннях сервера. Це може призвести до розкриття зайвих технічних деталей або слабкого захисту. Для виправлення додано security headers, CORS whitelist і єдиний формат помилок.

Сценарій A — SQL Injection

Суть уразливості

У функціоналі пошуку та фільтрації подій користувач передає дані через query-параметри. Такі дані вважаються недовіреними, тому їх не можна напряму вставляти в SQL-запит як частину рядка.

Небезпечним підходом було б формування SQL-запиту через конкатенацію рядка. Наприклад, якщо значення search напряму вставляється в SQL, користувач може передати не звичайний текст, а SQL-умову. У такому випадку значення search може змінити логіку SQL-запиту.

Відтворення

Для перевірки можна використати запит:

GET /api/v1/tasks?search=' OR '1'='1&sortBy=date&sortDir=desc&page=1&pageSize=5

Заголовок:

X-Demo-UserId: user-1

Такий ввід має сприйматися як звичайний текст, а не як частина SQL-команди.

Виправлення

У файлі backend/src/services/tasks.service.ts пошук виконується через параметризований запит. Значення користувача передається окремо як параметр, а не вставляється напряму в SQL-рядок.

Для сортування використано allowlist, тому користувач може передати тільки дозволені значення: title, date, location або capacity.

Перевірка

Після виправлення проблемний ввід не змінює структуру SQL-запиту. Якщо передати невалідне значення для sortBy, наприклад id; DROP TABLE Tasks, backend повертає помилку 400 VALIDATION_ERROR.

Легітимний сценарій також працює. Якщо виконати звичайний пошук за назвою події, сервер повертає список подій або порожній список без помилки.

Сценарій Б — XSS

Суть уразливості

У проєкті користувач вводить текстові дані для подій: назву, дату, локацію та інші поля. Якщо такі дані відображати на сторінці як HTML, браузер може сприйняти їх не як текст, а як розмітку або скрипт.

Приклад небезпечного введення:

Lab05 XSS <img src=x onerror=alert('xss')>

Якщо frontend вставляє таке значення через innerHTML, браузер може інтерпретувати цей рядок як HTML.

Відтворення

Для перевірки створюється подія з назвою, яка містить HTML-конструкцію.

Приклад body для Postman:

{
"title": "<b>XSS</b>",
"date": "2027-05-10",
"location": "Zaporizhzhia",
"capacity": 50
}

Запит виконується методом POST на маршрут:

POST /api/v1/tasks

Заголовки:

X-Demo-UserId: user-1
Content-Type: application/json

Якщо дані відображаються небезпечно, тег <b> може бути сприйнятий браузером як HTML, а не як текст.

Виправлення

У frontend дані користувача відображаються безпечно через DOM API та textContent. Такий підхід використовується для таблиці подій і таблиці статистики. Дані користувача показуються як звичайний текст, а не як HTML-код.

Перевірка

Після створення події з XSS-payload текст відображається у таблиці як звичайний рядок. Наприклад, користувач бачить саме текст <b>XSS</b>, а не жирний напис XSS. DOM не змінюється, небажаний JavaScript не виконується.

Також для полів Назва і Локація додано обмеження довжини, щоб дуже довгий текст не ламав вигляд таблиці.

Легітимний сценарій також працює. Якщо створити звичайну подію з нормальною назвою, вона створюється і коректно відображається у таблиці.

Сценарій В — Broken Access Control / IDOR

Суть уразливості

У проєкті події належать конкретним користувачам. Якщо доступ до події виконувати тільки за її id, без перевірки власника, інший користувач може підставити чужий id у URL і отримати доступ до чужої події.

Небезпечна логіка полягає в тому, що сервер шукає подію тільки за id. У такому випадку перевіряється лише існування події, але не перевіряється її власник.

Відтворення

Спочатку потрібно створити подію від імені користувача user-2.

Запит:

POST /api/v1/tasks

Заголовки:

X-Demo-UserId: user-2
Content-Type: application/json

Body:

{
"title": "Приватна подія user-2",
"date": "2027-07-20",
"location": "Локація user-2",
"capacity": 25
}

Після створення події потрібно скопіювати її id.

Далі потрібно виконати запит від імені іншого користувача:

GET /api/v1/tasks/<id-події-user-2>

Заголовок:

X-Demo-UserId: user-1

Якщо перевірки власника немає, користувач user-1 може отримати чужий ресурс.

Виправлення

На backend додано перевірку власника ресурсу. Для чутливих операцій використовується умова, яка перевіряє не тільки id події, а й userId поточного користувача.

Захищені операції:

GET /api/v1/tasks/:id
PATCH /api/v1/tasks/:id
PUT /api/v1/tasks/:id
DELETE /api/v1/tasks/:id

Також за поточним користувачем фільтруються списки подій, статистика, count, top, by-date та with-users.

Перевірка

Якщо користувач намагається отримати чужу подію, backend повертає 404 NOT_FOUND.

Приклад відповіді:

{
"success": false,
"status": 404,
"code": "NOT_FOUND",
"message": "Task not found"
}

Якщо користувач user-1 намагається змінити або видалити подію user-2, сервер також повертає 404 NOT_FOUND.

Легітимний сценарій працює. Якщо користувач user-2 відкриває свою подію, сервер повертає відповідь 200 OK і дані цієї події.

Сценарій Г — Security Misconfiguration

Суть проблеми

Навіть якщо основна логіка проєкту працює правильно, вебзастосунок може бути вразливим через неправильні налаштування сервера. До таких проблем належать відсутність базових security headers, занадто широкий CORS, зайві технічні деталі у відповідях з помилками та неузгоджені статус-коди.

Відтворення

Для перевірки можна виконати запит без заголовка X-Demo-UserId:

GET /api/v1/tasks

Якщо налаштування помилок неправильні, сервер може повернути зайві технічні деталі, stack trace або внутрішні шляхи до файлів.

Виправлення

У проєкті додано базові security headers:

X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: no-referrer

CORS обмежено локальними frontend-origin:

http://localhost:5500
http://127.0.0.1:5500
http://localhost:5173
http://127.0.0.1:5173

Також використовується єдиний формат помилок.

Приклад відповіді:

{
"success": false,
"status": 401,
"code": "UNAUTHORIZED",
"message": "Unauthorized",
"details": []
}

Перевірка

Запит без заголовка X-Demo-UserId повертає 401 Unauthorized. Сервер не розкриває зайві технічні деталі. У відповідях присутні базові security headers.

Також перевіряється легітимний сценарій. Якщо виконати запит із правильним заголовком X-Demo-UserId: user-1, сервер повертає список подій або порожній список без помилки.

Статистика TaskStats

У проєкті використовується окрема таблиця статистики TaskStats. Вона потрібна для збереження та відображення агрегованих даних про події.

Статистика включає найдовшу назву події, довжину найдовшої назви, найбільшу місткість події, назву події з найбільшою місткістю, кількість майбутніх подій, кількість минулих подій, розподіл подій за місяцями та дату оновлення статистики.

Цей функціонал дозволяє не тільки працювати з подіями, а й аналізувати їх у вигляді окремої статистичної таблиці.

Перевірка TaskStats

Для перевірки статистики потрібно виконати запит:

GET /api/v1/tasks/stats

Заголовок:

X-Demo-UserId: user-1

Очікуваний результат: сервер повертає статистику для поточного користувача. У відповіді мають бути поля longestTitle, longestTitleLength, biggestCapacity, biggestCapacityTitle, upcomingEvents, pastEvents, byMonth та updatedAt.

Додаткова перевірка основного функціоналу

Для створення події використовується POST /api/v1/tasks з коректним заголовком X-Demo-UserId.

Для отримання списку використовується GET /api/v1/tasks.

Для оновлення своєї події використовується PATCH /api/v1/tasks/<id>.

Для видалення своєї події використовується DELETE /api/v1/tasks/<id>.

Для перевірки кількості подій використовується GET /api/v1/tasks/count.

Для перевірки топ-3 подій використовується GET /api/v1/tasks/top.

Для перевірки подій з користувачами використовується GET /api/v1/tasks/with-users.

Для перевірки статистики використовується GET /api/v1/tasks/stats.

Висновок

У межах лабораторної роботи №5 у проєкті Project_OAP реалізовано захист від основних типових уразливостей вебзастосунків. Для SQL Injection використано параметризовані запити та allowlist, для XSS — безпечне відображення даних, для IDOR — перевірку власника ресурсу на backend, а для Security Misconfiguration — базові security headers, обмежений CORS і єдиний формат помилок.

Основний функціонал проєкту працює: CRUD подій, пошук, сортування, пагінація, топ-3 подій, count, with-users та таблиця статистики TaskStats.
